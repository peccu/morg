### T001 · i18n対応

| 項目 | 内容 |
|------|------|
| **ステータス** | 🚧 対応中 |
| **優先度** | 中 |
| **概要** | UI文字列の国際化対応（日本語・英語） |

**調査内容**
- 現状: 日本語ハードコード
- ライブラリ: vue-i18n
- 言語: 日本語・英語
- 言語選択: ブラウザ設定に自動追従 + UI切替ボタン + localStorage保存

**対応方針**
1. vue-i18n をインストール
2. `src/i18n/` に `ja.ts` / `en.ts` ロケールファイルを作成
3. `src/i18n/index.ts` で i18n インスタンスを生成（localStorage + ブラウザ検出）
4. `main.ts` に登録
5. 全コンポーネントの日本語文字列を `$t()` / `t()` に置き換え
6. InboxView のヘッダーメニューに言語切替ボタンを追加

**確認方法**
- ブラウザ言語設定を英語にして開くと英語UIが表示される
- UI切替で即時反映される
- 切替後リロードしても言語が維持される

### T004 · ランディングページ（未ログイン時のトップ画面）

| 項目 | 内容 |
|------|------|
| **ステータス** | 📋 対応予定 |
| **優先度** | 中 |
| **概要** | 未ログイン時のログイン画面を、アプリの説明・売りが伝わるランディングページに刷新する |

**背景**
- 現状の `/login` は Google ログインボタンだけのシンプルな画面
- 初めて訪れたユーザーが「何のアプリか」「使う価値があるか」を判断できない
- OSS として公開しているため、GitHubから来た人にも伝わる内容にしたい

**調査内容**
- 現状ファイル: `apps/web/src/views/LoginView.vue`
- 伝えたいこと（案）:
  - Gmail をスマホで快適に整理するためのPWAアプリ
  - 一括アーカイブ・削除・既読などの操作がモバイルで快適
  - プラグインで拡張可能（Webhook送信など）
  - OSSで自分でデプロイして使える

**対応方針**
- `LoginView.vue` をランディングページ兼ログイン画面として拡張
- 構成案: ヒーロー（キャッチコピー + Google ログインボタン）→ 機能紹介セクション → フッター（GitHub リンク）
- デザインは既存の forest カラーパレットに合わせる
- モバイルファーストで縦スクロール型

**確認方法**
- 未ログイン状態で `/login` を開き、アプリの概要・機能・ログインボタンが表示されること
- モバイル幅（375px）で崩れないこと

### T003 · invalid_grant 時の再ログイン誘導

| 項目 | 内容 |
|------|------|
| **ステータス** | ✅ 完了 |
| **優先度** | 高 |
| **概要** | `invalid_grant` 発生時に500ではなく401を返し、フロントがログイン画面に誘導するようにする |

**調査結果**
- リフレッシュトークン自体を別トークンで更新する手段はOAuth 2.0仕様上存在しない
- `invalid_grant` = リフレッシュトークンの死亡 → ユーザーの再ログインが必須
- アプリを定期的に使っていれば `getValidToken()` のリフレッシュ呼び出しが非アクティブタイマーをリセットするため、普通の利用では6ヶ月失効は起きにくい
- **本当の問題**: `invalid_grant` 時も `500` を返しており、フロントが再ログインを促せない
  - `packages/functions/src/lib/google-auth.ts:56` の `refreshAccessToken()` が失敗時に `throw new Error('Token refresh failed: ...')` するだけ
  - 呼び出し元の各 gmail function がそのまま500を返す

**対応方針**
1. `token.ts` の `getValidToken()` で `tokens.refresh_token` が返ってきた場合にセッションへ保存する（現状は無視している）
   - Googleはリフレッシュ時にまれに新しいリフレッシュトークンを返す場合があり、それを使い続けることで寿命を延ばせる
2. `google-auth.ts` の `refreshAccessToken()` で `invalid_grant` を検知し専用エラーを throw
3. `token.ts` の `getValidToken()` でそれを catch して `401` 相当のエラーとして再 throw
4. 各 gmail function で catch して `{ statusCode: 401 }` を返す
5. フロント（`useThreads` 等）で 401 を検知 → `useAuthStore().logout()` → ログイン画面へ

**確認方法**
- Google アカウント設定でアプリのアクセスを手動で取り消した後にAPIを叩く
- `500` ではなく `401` が返り、アプリがログイン画面に遷移すること
